# Аудит: /api/generate, pains, маппинг галочек

## 1. Где вызывается /api/generate

| Источник | Payload | Ожидаемый ответ |
|----------|---------|------------------|
| **GeneratorForm.tsx** (основной UI) | `{ category, audience, pain, comment, name, uniqueness, temperature, diagnostics, include, projectId }` | Полный **draft** паспорта: `{ header, blocks, tech?, star?, conclusion? }` |
| **public/questions.js** (legacy) | `{ form: payload }`, payload = `{ category, name, audience, pains[], uniqueness, temperature }` | Полный **draft**: используется `fillBrief(draft)`, `fillFull(draft)` с `draft.blocks`; `draft.pains[0]` — только fallback для поля «Боль» в кратком паспорте |

**Вывод:** оба клиента ожидают **полный паспорт**, а не только `{ pains }`.

---

## 2. Где используется ответ { pains }

- **api/generate.js:** `source.pains` — это **вход** из body (массив болей от пользователя), не ответ API.
- **GeneratorForm.tsx:** переменная `pains` — локальный state со списком **примеров** болей по возрасту (`painsByAge`), заполняется по кнопке «Показать примеры боли», к ответу API не привязана.
- **public/questions.js:** единственное использование ответа — `draft?.pains && draft.pains[0]` в `fillBrief()` как запасной вариант, если нет `header.pain`. То есть UI рассчитан на полный draft; `pains` в ответе — опциональное дополнение.

**Поиск по проекту:** нет ни одного места, где бы вызывался `/api/generate` с ожиданием **только** `{ pains }`. Режим «только боли» в текущем коде не используется.

**Доказательство:** в `app/api/generate/route.js` сейчас всегда вызывается `runGenerate(body)` и возвращается draft паспорта. Отдельного режима `mode: "pains"` нет, и ни один компонент (в т.ч. questions.js) не проверяет `mode` и не обрабатывает ответ как массив болей.

**Вывод:** режим pains не используется; оставляем только генерацию паспорта. Добавлять ветку `mode: "pains"` не требуется.

## 4. Что нужно исправить (кратко)

- **A) Маппинг:** d2 → `packaging` (и при необходимости отдельная секция/ключ `packaging` в API и JSON). Включить в include ключи: `1.1`..`4.5`, `tech`, `packaging`, `star`, `conclusion`; header.* по желанию (сейчас не управляются галочками — ок).
- **B) Prompt:** уже строится от include (enabledSchemas, финальные секции); явно запрещать отключённые секции — уже есть.
- **C) JSON:** normalizeDraft/buildFallbackDraft уже возвращают только включённое; добавить поддержку `packaging` (поле в draft и в include).
- **D) UI:** ResultPreview уже рендерит только пришедшие блоки и секции; убедиться, что `packaging` отображается при наличии.
- **E) Стиль Шоковсянка:** смягчить формулировку: «предпочитай связные абзацы 2–5 предложений; допускай короткие перечисления только если это улучшает ясность; запретить слова варианты/предлагаем/я рекомендую».

1. **Шаг 1.** Добавить ключ `packaging` в include (фронт + бэкенд), маппинг d2 → packaging; в API — prompt/нормализация/fallback для `packaging`. Без новых роутов.
2. **Шаг 2.** Смягчить в промпте правило про списки (стиль Шоковсянка по п. E).
3. **Шаг 3.** В ResultPreview показывать секцию «Упаковка/форм-фактор» только если в draft есть `packaging`.
4. **Шаг 4.** Ручная проверка: выключить 2.3, 4.5, packaging → их нет в prompt/JSON/UI; включить только 1.1 + tech → только они есть; при отсутствии include — всё включено.

Режим pains не восстанавливаем — не используется.

## 6. Ручная проверка (после правок)

1. **Выключить 2.3 и 4.5:** в блоке «Пожелания к новому продукту» для пунктов 2.3 (обонятельный образ) и 4.5 (Продвижение) нажать «Нет». Создать продукт. Ожидание: в ответе API и в превью нет пунктов 2.3 и 4.5; в промпте (логи сервера) их нет.
2. **Выключить packaging:** в «Дополнительно» для d2 («форм-факторы и упаковка») нажать «Нет». Создать продукт. Ожидание: в JSON нет поля `packaging`, в превью нет секции «Форм-фактор и упаковка».
3. **Только 1.1 + tech:** оставить «Да» только для c1 (1.1) и d1 (tech), остальное «Нет». Создать продукт. Ожидание: в ответе только header, блок cognitive с одним пунктом 1.1, и секция tech; в превью только они.
4. **Без include (обратная совместимость):** запрос без поля `include` (например, из questions.js или Postman). Ожидание: все секции включены, полный паспорт.
